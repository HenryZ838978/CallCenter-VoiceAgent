风险项
  • ⚠ RAG Prompt 膨胀 — 检索 context 过多会拖慢 TTFT，严格限制 Top-K=1~2，单 chunk ≤ 200 token
  • ⚠ AutoDL 网络限制 — 不支持公网 SIP/RTP，必须经腾讯 VPS 中转，隧道稳定性需要监控
  • ⚠ VoxCPM INT8 精度 — 扩散模型量化到 INT8 可能音色失真，TTS 必须保持 FP16
  • ⚠ 马来语 ASR 准确率 — FunASR 对 Manglish / 混语 code-switching 的表现待验证
  • ⚠ Scicom + TM 合作 — Scicom 已 18 个月开发自有 AI + Telekom Malaysia 战略合作，若先覆盖 VADS 政企客户则堵死入口
  • ⚠ Azure 马来区开服 — 微软 Azure AI Speech 若推出 Manglish 音色，本地部署 + 口音壁垒将被削弱
  • ⚠ LLM 不输出语气词 — TTS 无法凭空插入 "lah/meh/ah"，若 LLM 端 Manglish 能力不足则口音再好也像"念稿"而非"聊天"
  • ⚠ 开源模型追赶 — Llama 4 / Qwen 3 持续降低复制门槛，纯技术栈差异化窗口期有限，需用数据飞轮构建持久壁垒

Debug 纪实 (2026-03-03)

从零搭建到 v2.0 全过程中遇到并解决的关键问题

TTS 输出像噪声 — 采样率 Bug
VoxCPM VAE 输出 44100Hz，代码硬编码 24000Hz → 播放速度降为 54% → 严重失真。修复: SAMPLE_RATE = 44100
Voice Clone 质量差 — 温度过高
默认 temp=1.0 导致多人抢话感。AB 测试 4 组参数后确认 temp=0.7, cfg=3.0 最优。
每段音频播放前爆音 — 播放架构缺陷
BufferSourceNode 每次 start() 产生 transient click。经历 3 轮修复: fade-in/out → nextPlayTime 排队 → 最终改为 AudioWorklet Queue Player (持续输出，零爆音)。同时 VoxCPM 首 chunk (160ms) 含 VAE startup transient，直接丢弃。
torchcodec ABI 不兼容
nanovllm 子进程内 torchaudio.load → torchcodec → FFmpeg ABI 冲突。通过 usercustomize.py 全局 patch torchaudio.load 为 soundfile 实现。
Event Loop Already Running
nanovllm SyncPool 的 run_until_complete 与 uvicorn 事件循环冲突。v1 通过预加载(load before uvicorn)规避；v2 句级流式中 LLM producer 线程与 TTS 线程并发触发，通过 nest_asyncio.apply() 解决。
LLM 重复生成 + 念 think 标签
MiniCPM4.1 enable_thinking=False 仍偶尔输出 think 块。3 层正则清洗 + repetition_penalty=1.2 + max_tokens=150 解决。
ASR 识别成日语
SenseVoice language="auto" 在低音量时误判语种。强制 language="zh" 解决。
打断后 TTS 仍念完 — 流式取消粒度
v1: list() 一次收集所有句子，cancel 不及时。v2: asyncio.Queue 真正逐句流式 + 200ms 小块发送 + 浏览器 ignoreAudio flag + AudioContext flush。

v2.0 问题诊断 (6 项)

P0 致命: 前端 SPEAKING 停发音频
agentState !== 'speaking' 阻止麦克风音频发送 → 服务端 SPEAKING 状态永远收不到音频 → 打断完全失效
P0: 单帧触发打断
32ms 单个 chunk speech_prob ≥ 0.5 即触发 → 噪声/回声/咳嗽 都会误打断
P1: THINKING 丢弃用户语音
Pipeline 运行 ~458ms 期间用户追加的内容被 pass 忽略
P2: 固定 256ms Endpointing
用户停顿思考 → 系统抢答；不区分短语/长句场景
P2: TTS 取消粒度 200ms
打断后仍残留 200ms 音频播放
P3: ignoreAudio 状态泄漏
barge_in 置 true，仅在 filler (已禁用) 时清除 → 永远 true
